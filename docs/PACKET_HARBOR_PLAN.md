# PACKET HARBOR 実装計画

## 目的

固定アニメーションを中心としたOSI解説から、来場者の操作が実際の通信状態を変える展示へ移行する。

- Raspberry Pi A: 正常通信と負荷通信を発生させる
- Raspberry Pi B: HTTPサービスとXDP/eBPFを動かす
- 展示画面: 実測したPASS/DROP、応答時間、正常通信成功率だけを表示する
- 物理筐体: 防御モードを切り替える操作部と通信状態のフィードバックを持つ

## 守る原則

1. 観測できていない出来事を観測済みとして表示しない
2. タイマーではなく実イベントで画面状態を進める
3. 制御用通信と展示対象通信を明確に分離する
4. 正常系と攻撃系を同時に流し、防御後も正常通信が通ることを示す
5. 1回の体験を30秒以内に収める

## Phase 1: 観測の正しさ

- 物理操作イベントに期待プロトコルと宛先ポートを付与する
- 送信元IPと時刻だけの相関を廃止する
- XDPの実測相関が完了するまで展示を開始しない
- 誤相関を再現するテストを追加する

## Phase 2: XDP防御

- monitorとprotectの2モードを追加する
- UDP負荷通信専用ポートを定義する
- protect時は負荷通信をXDP_DROPする
- PASS/DROPカウンタをper-CPU mapで集計する
- 全パケットをRingBufへ流さず、サンプリングと集計イベントを分ける

## Phase 3: 2台のRaspberry Pi

### Pi A: traffic-node

- TCP/HTTPの正常なヘルスチェックを継続する
- 展示操作時だけUDP負荷通信を発生させる
- 成功率と応答時間を計測する

### Pi B: harbor-node

- HTTPサービスを提供する
- XDPをNICへattachする
- 物理スイッチから防御モードを変更する
- 観測結果をdashboardへ配信する

## Phase 4: 展示UI

表示する値は以下に限定する。

- 正常通信の成功率
- HTTP応答時間
- 受信パケット数
- XDP_PASS数
- XDP_DROP数
- 現在の防御モード

OSI 7層の自動再生は廃止する。詳細解説は任意で開く補助画面へ移す。

## Phase 5: 物理筐体

3Dプリンタで港の検問所を模した筐体を作る。

- 中央: 防御モード切替レバー
- 左: 正常通信を示す経路
- 右: 負荷通信を示す経路
- 背面: Raspberry Piと配線を交換できる保守用パネル

筐体は装飾ではなく、操作とシステム状態を対応させるインターフェースとして設計する。

## 完成条件

1. 負荷通信開始後に正常通信の応答悪化を実測できる
2. protectへ切り替えるとXDP_DROPが増える
3. protect中も正常HTTP通信は成功する
4. 画面の全数値を実測イベントから追跡できる
5. ネットワークを切っても誤って成功演出へ進まない
