## パケットを分類する考え方

XDP は NIC に入ってきたパケットの先頭から読む。今回読んだのはまず Ethernet ヘッダと IPv4 ヘッダ。

```text
Ethernet ヘッダ: 14 バイト
  0-5    宛先 MAC アドレス
  6-11   送信元 MAC アドレス
  12-13  EtherType

IPv4 ヘッダ:
  14     version / IHL
  23     protocol
  26-29  source address
  30-33  destination address
```

EtherType が `0x0800` なら IPv4。IPv4 の `protocol` が `1` なら ICMP、`6` なら TCP、`17` なら UDP。

最初は全パケットでこれだけだった。

```rust
info!(&ctx, "received a packet");
Ok(xdp_action::XDP_PASS)
```

今は Ethernet / IPv4 / TCP / UDP / ICMP のフィールドを読んで、ログを分けている。

## eBPF で気をつけたこと

パケットを読む前に必ず `data_end` で範囲チェックする必要がある。範囲外を読む可能性があると verifier に弾かれる。

今回入れた `ptr_at` は、`offset` の位置に指定サイズのデータが本当にあるかを確認してからポインタを返すための関数。

```rust
fn ptr_at<T>(ctx: &XdpContext, offset: usize) -> Result<*const T, u32> {
    let start = ctx.data();
    let end = ctx.data_end();
    let len = mem::size_of::<T>();

    if start + offset + len > end {
        return Err(xdp_action::XDP_ABORTED);
    }

    Ok((start + offset) as *const T)
}
```

一度、IPv4 ヘッダ長（IHL）から transport header の開始位置を動的に計算して TCP/UDP ポートを読もうとしたが、verifier が安全性を追い切れず `invalid access to packet` でロードに失敗した。

今回は学習の段階なので、**IPv4 options なし（IHL=5）の通常パケットだけ詳細を読む**形にした。options 付きは `received IPv4 packet with options` とログを出して通す。

```text
IHL=5
=> IPv4 ヘッダは 20 バイト
=> TCP/UDP/ICMP ヘッダ開始位置は Ethernet 14 + IPv4 20 = 34 バイト目
```

この固定オフセットなら verifier を通しやすく、実際にロードとログ確認まで通った。

## L7 の入り口を見る

TCP の payload 先頭を読んで、HTTP の `GET ` / `POST` を検知するところまで進めた。

TCP ヘッダは必ず 20 バイトとは限らない。TCP options があると長くなるので、TCP ヘッダの 12 バイト目にある Data Offset を読む。

```text
Data Offset = 5  => TCP ヘッダ 20 バイト
Data Offset = 6  => TCP ヘッダ 24 バイト
...
Data Offset = 15 => TCP ヘッダ 60 バイト
```

payload の開始位置はこうなる。

```text
Ethernet 14 + IPv4 20 + TCP header length
```

ただし verifier は動的 offset に厳しい。そこで `5..=15` を `match` で分けて、payload offset が明示的な値になるようにした。

payload 先頭4バイトを読んで、

```text
GET␠
POST
```

ならログを出す。

確認には、一時的な veth と network namespace を作った。HTTPS では payload が暗号化されて `GET` / `POST` が見えないので、平文 HTTP サーバーへアクセスする必要がある。

確認できたログ。

```text
detected HTTP GET: src=10.10.0.2:58344, dst=10.10.0.1:8081
```

ここでできたのは「HTTP の中身全部を見る」ではなく、**TCP payload の先頭が HTTP メソッドっぽいかを見る**ところまで。

## RingBuf で userspace に事実を渡す

ログ文字列だけだと、人間には読めるが、あとで RocksDB に保存するには扱いづらい。そこで、eBPF 側で解析した 5-tuple を構造体にして userspace 側へ渡すようにした。

共通 crate に `FlowEvent` を置いた。

```rust
#[repr(C)]
pub struct FlowEvent {
    pub src_addr: [u8; 4],
    pub dst_addr: [u8; 4],
    pub src_port: u16,
    pub dst_port: u16,
    pub protocol: u8,
    pub _pad: [u8; 3],
}
```

`#[repr(C)]` は eBPF 側と userspace 側でメモリ上の並びを合わせるため。`_pad` は構造体サイズとアラインメントを安定させるために入れている。

eBPF 側では `EVENTS` という RingBuf map を作った。

```rust
#[map]
static EVENTS: RingBuf = RingBuf::with_byte_size(1 << 20, 0);
```

パケットを解析したら `FlowEvent` を reserve → write → submit する。

```text
RingBuf に FlowEvent 1個分の場所を予約
↓
5-tuple を書く
↓
submit して userspace に見えるようにする
```

userspace 側では `EVENTS` を `take_map` して `RingBuf` として開き、`AsyncFd` で読み取り可能になるのを待つ。受け取った bytes を `FlowEvent` に戻して表示する。

確認できたログ。

```text
flow event: proto=TCP, src=10.10.0.2:60838, dst=10.10.0.1:8081
```

これで「eBPF が見た事実」を userspace の Rust が構造体データとして扱えるようになった。RocksDB に保存するなら、この `FlowEvent` をそのままキー/値に変換する流れにできる。

## 決定論的な異常検知

次に、単純な統計ベースの異常検知を eBPF 側に入れた。

ルールはこれ。

```text
1秒間に同じ宛先IPへ100パケット以上来たら warn
```

宛先IPごとの状態を持つために `PACKET_WINDOWS` という HashMap を追加した。

```rust
struct PacketWindow {
    window_start_ns: u64,
    count: u32,
    warned: u32,
}
```

意味はこう。

```text
window_start_ns: その1秒窓の開始時刻
count:           窓内で数えたパケット数
warned:          その窓で warn を出したか
```

時刻は `bpf_ktime_get_ns()` で取る。1秒は `1_000_000_000` ns。

処理の流れ。

```text
パケットが来る
↓
dst IP を u32 キーにする
↓
PACKET_WINDOWS で状態を見る
↓
初めての宛先なら count=1 で作る
↓
同じ1秒窓なら count += 1
↓
count >= 100 かつ warned == 0 なら warn
↓
warned=1 にして同じ窓では何度も warn しない
```

確認では namespace 側から `10.10.0.1` へ短時間で 120 発 ping を流した。

```text
[WARN  xdp_hello] high packet rate: dst=10.10.0.1, count=100 in 1s
```

これで「事実の観測」から「決定論的な異常判定」までつながった。
