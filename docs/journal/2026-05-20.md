## パケットを分類する考え方

XDP は NIC に入ってきたパケットの先頭から読む。今回読んだのはまず Ethernet ヘッダと IPv4 ヘッダ。

```text
Ethernet ヘッダ: 14 バイト
  0-5    宛先 MAC アドレス
  6-11   送信元 MAC アドレス
  12-13  EtherType

IPv4 ヘッダ:
  14     version / IHL
  23     protocol
  26-29  source address
  30-33  destination address
```

EtherType が `0x0800` なら IPv4。IPv4 の `protocol` が `1` なら ICMP、`6` なら TCP、`17` なら UDP。

最初は全パケットでこれだけだった。

```rust
info!(&ctx, "received a packet");
Ok(xdp_action::XDP_PASS)
```

今は Ethernet / IPv4 / TCP / UDP / ICMP のフィールドを読んで、ログを分けている。

## 実際に見えたログ

Lima の SSH 系の通信は TCP として見えた。

```text
received IPv4 TCP packet: src=192.168.5.2:18072, dst=192.168.5.15:22
```

`dst=...:22` なので、VM の SSH ポートに向かう通信だと読める。

`ping` を流すと ICMP として見えた。

```bash
limactl shell ubuntu-lts -- ping -c 1 8.8.8.8
```

確認できたログ。

```text
received IPv4 ICMP packet: src=8.8.8.8, dst=192.168.5.15, type=0
```

ICMP type `0` は echo reply。つまり `8.8.8.8` から VM に返ってきた ping 応答を XDP が拾っている。

## eBPF で気をつけたこと

パケットを読む前に必ず `data_end` で範囲チェックする必要がある。範囲外を読む可能性があると verifier に弾かれる。

今回入れた `ptr_at` は、`offset` の位置に指定サイズのデータが本当にあるかを確認してからポインタを返すための関数。

```rust
fn ptr_at<T>(ctx: &XdpContext, offset: usize) -> Result<*const T, u32> {
    let start = ctx.data();
    let end = ctx.data_end();
    let len = mem::size_of::<T>();

    if start + offset + len > end {
        return Err(xdp_action::XDP_ABORTED);
    }

    Ok((start + offset) as *const T)
}
```

一度、IPv4 ヘッダ長（IHL）から transport header の開始位置を動的に計算して TCP/UDP ポートを読もうとしたが、verifier が安全性を追い切れず `invalid access to packet` でロードに失敗した。

今回は学習の段階なので、**IPv4 options なし（IHL=5）の通常パケットだけ詳細を読む**形にした。options 付きは `received IPv4 packet with options` とログを出して通す。

```text
IHL=5
=> IPv4 ヘッダは 20 バイト
=> TCP/UDP/ICMP ヘッダ開始位置は Ethernet 14 + IPv4 20 = 34 バイト目
```

この固定オフセットなら verifier を通しやすく、実際にロードとログ確認まで通った。
